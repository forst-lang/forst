You write **`.ft`** files with structural typing and validation at the boundary. The compiler emits Go you build with `go build`. You get less boilerplate at the HTTP boundary and in domain logic.

<CatalogOrder />

## How it fits your stack

```mermaid
flowchart LR
  ft[".ft sources"]
  goOut["Generated Go"]
  deploy["go build / deploy"]
  dts["generated/types.d.ts"]
  clients["API clients"]

  ft --> goOut --> deploy
  ft -.-> dts -.-> clients
```

Run `forst generate` when clients need TypeScript types from the same source.

## Who this is for

For teams that want Go's runtime with less validation boilerplate and stronger types.

<Info>
  Forst is actively developed. Some features (Result types, Go import loading, sidecar) are **experimental**. See the [roadmap](/resources/roadmap) for current status.
</Info>
