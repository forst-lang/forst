You write **`.ft`** files with structural types and validation at the boundary. The compiler emits Go you build with `go build`. You get less boilerplate at the HTTP boundary and in domain logic.

## How it fits your stack

```mermaid
flowchart LR
  ft[".ft sources"]
  goOut["Generated Go"]
  deploy["go build / deploy"]
  dts["generated/types.d.ts"]
  clients["API clients"]

  ft --> goOut --> deploy
  ft -.-> dts -.-> clients
```

You define shapes and handlers in Forst. The compiler validates them and emits Go for production. When you need shared client types, run `forst generate` to emit declarations from the same source.

## Who this is for

You want **Go's runtime and deployment** with **less validation boilerplate** and **stronger types**, whether you start fresh or mix Forst into an existing module.

<Info>
  Forst is actively developed. Some features (Result types, Go import loading, sidecar) are **experimental**. See the [roadmap](/resources/roadmap) for current status.
</Info>
