Forst transpiles to **Go**. You keep the same module graph, deployment, and ecosystem. See [Language overview](/language/overview) for syntax. This page covers import and mixing rules.

## Transpile to idiomatic Go

Every `.ft` file compiles to Go source in your module. The output is an ordinary, readable Go source file. Struct fields, error returns, and control flow follow familiar Go patterns.

```bash
npx forst build handlers/orders.ft
```

Run the result with the standard toolchain:

```bash
go build ./...
go test ./...
```

## Go shaped syntax

Much of Forst mirrors Go syntax: `package`, `import`, `func`, `if`/`for`/`range`, `defer`, `go` goroutines. A minimal program can be identical in Forst and Go. See [Language overview](/language/overview).

Forst adds structural typing, constraints, `ensure`, and nominal errors on top of that base.

## Import Go packages

Call into the standard library and third-party modules with qualified names:

<InteropFmtHello />

The compiler loads Go packages via **`go/packages`** and type-checks ForstтЖФGo calls when imports resolve. Supported mappings today include primitives, slices, pointers, `error`, and `interface{}` (including variadic calls).

Go functions that return **`(T, error)`** map to **`Result(T, Error)`** in Forst at call sites. That applies to the **result type** of the call. It applies when a trailing expression becomes an [implicit return](/language/errors-and-result#return-inference-and-implicit-returns). This differs from using **`println`** / **`print`** for void side effects inside a function that should not return a value.

<Warning>
  **Go import loading is experimental.** Unmapped Go types report diagnostics; `unsafe` and some builtin edge cases are not fully wired. Put `unsafe` usage in hand written `.go` and call from Forst when needed.
</Warning>

## Mixed packages

Place `.ft` files alongside `.go` in the same module:

```text
my-service/
тФЬтФАтФА go.mod
тФЬтФАтФА handlers/
тФВ   тФЬтФАтФА orders.ft
тФВ   тФФтФАтФА legacy.go
тФФтФАтФА internal/
    тФФтФАтФА db.go
```

Use **`-root`** to merge all same-package sources under a tree, matching discovery for `forst run`, `forst build`, and the dev server:

```bash
npx forst run -root ./handlers handlers/orders.ft
```

The typechecker walks up from your `.ft` file to find **`go.mod`** (`GoWorkspaceDir`).

## Runtime validation

Type constraints in Forst emit **runtime checks** in generated Go. Untrusted input is validated at the boundary before your handler body executes. See [Shapes and constraints](/language/shapes-and-constraints).

## Providers

Forst **Providers** (`use` / `with`) declare shared runtime services at function boundaries and wire implementations at entry points. See [Providers (DI)](/language/providers).

```bash
task example:providers
task example:providers-cross_pkg
```

## When to keep hand written Go

- **`unsafe`** or low-level syscalls not yet mapped
- Large existing packages you are migrating incrementally
- Performance-critical paths already audited in Go

Forst supports **gradual adoption**. You can migrate route by route without a big bang rewrite.
